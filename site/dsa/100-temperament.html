Problem Solving Temperament

LIVING DOCUMENT · add entries as you observe patterns in your daily practice

These are not tricks. They are corrections to mental habits that degrade under pressure. Read before a contest, revisit when you make a mistake you've made before.

Under Pressure
01 Fear makes you skip exactly the step that matters
Root cause: time anxiety

When you're nervous, there's a temptation to half-check something — to glance at the indexing and tell yourself "it probably works." It almost never does. The exact thing you skimmed is where the bug lives.

The counter-intuitive truth: taking 90 seconds to trace through a small example is faster than submitting, getting WA, re-reading for 5 minutes in a panic, and submitting again. The "shortcut" costs more time, not less.

Example
Off-by-one in a two-pointer — you convince yourself the boundary looks fine. It isn't. Take two concrete values, run your loop condition by hand. Two minutes of dry-run beats three wrong submissions.
02 Always test the edge case you're most tempted to skip
Root cause: optimism bias

The cases worth testing before submitting: empty input, single element, all-same elements, the last/first index, when the answer is exactly at a boundary. These aren't exotic — they're the first tests any judge runs.

Checklist before submitting
Empty case · single element · first/last element as answer · duplicate values · negative numbers (if applicable) · the loop termination condition at n-1
Clarity of Thought
03 Do not invent constraints the problem didn't give you
Root cause: over-engineering

Re-read the problem statement and ask for every constraint you're about to code around: did the problem actually say this? A lot of complexity comes from protecting against things that aren't required.

Example
Problem: remove all elements equal to val, order doesn't matter, return new length. You don't need to preserve the rest of the array — overwriting is fine. You don't need a separate result array. Two lines with two pointers, done. The instinct to "be careful" about the values you're overwriting is protecting against a constraint that doesn't exist.
04 For every line of code: "do I actually need this?"
Root cause: defensive coding

Before writing a variable, a condition, a pointer — ask so what? What goes wrong if I don't have this? If nothing goes wrong, don't write it. Unnecessary variables are not just noise; they're things to maintain, things to get wrong, things to confuse yourself with under pressure.

Example
Browser history (linked list problem): almost every solution uses a curr pointer alongside head. But if you're only ever moving forward from the front, head itself is the cursor — moving head = head.next is identical and cleaner. The curr variable exists because it feels safer, not because it's needed.
05 Simple solutions are usually the correct ones
Mindset: minimalism

If your solution has grown to 30 lines and involves multiple auxiliary structures, pause. A simpler solution almost certainly exists. Complexity is a sign you haven't found the right framing yet — not a sign of rigor.

The best solutions are short because they match the structure of the problem. When you find that structure, the code writes itself.

Heuristic
If you're done and your solution is longer than you'd expect for the difficulty level — re-read the problem. You likely assumed a constraint that isn't there, or missed a property that makes things simpler.
Process
06 Understand the output before you think about the algorithm
Root cause: jumping ahead

Visualise exactly what the output should look like for two or three concrete inputs — before thinking about how to produce it. Problems feel hard because the algorithm seems unclear; usually the algorithm becomes obvious once the expected transformation is crystal clear.

Practice
Write out: input → expected output, for at least two examples (one normal, one edge). Then ask: what did I have to do mentally to get from one to the other? That's usually the algorithm.
07 Slow down at the transitions — beginning, end, direction change
Root cause: autopilot in the middle

Bugs live at the start of a loop, at the termination condition, and at any point where logic changes direction (e.g. switching from left-to-right to right-to-left, or from expansion to contraction in a sliding window). The middle of a loop is usually fine. Slow down at the boundaries.

The Two Non-Negotiables
08 Always start from brute force
Non-negotiable

Never jump straight to the optimal. Start by thinking through the naive solution — even if you'd never submit it. This isn't wasted time. It forces you to fully understand what you're computing, catches constraints you glossed over when reading, and almost always reveals the structure that the optimisation exploits.

Brute force is also your settlement time. The problem that seemed vague when you first read it becomes concrete once you've thought through even a slow solution. You arrive at the optimisation phase with a clear head rather than a panicked one.

Why it works
Most optimal solutions are brute force with one observation applied — a redundant loop collapsed with a hashmap, a repeated computation memoized, a linear scan replaced with a sorted structure. You can only see the observation if you've first seen the brute force.
09 Write pseudo code before writing code
Non-negotiable

If your pseudo code doesn't work, your code will not work. If your pseudo code works, the code will work — and it will take only a few minutes to write, because all the decisions are already made.

This feels like spending time. It is the opposite. The speed of your mind is effectively infinite — once you know exactly what to do, implementation is mechanical. All the real time is spent deciding: what structure, what order, what condition, what the loop invariant is. Pseudo code is where that deciding happens, cleanly, without the noise of syntax.

Never start typing code while there is still any open question about the approach. Every line of code written before the approach is settled is likely to be rewritten.

What pseudo code looks like
Not formal syntax. Just: "use two pointers l, r. while l < r, if sum < target move l right, if sum > target move r left, if equal record and move both." That's enough. If you can write that and believe it, the code is ten minutes away. If you can't write that, no amount of typing will save you.
