Linear Data StructureStack

A stack is defined by three operations — and the constraint that all three must be O(1):

OperationDescriptionComplexity
push(x)Add element to topO(1)
pop()Remove + return top elementO(1)
peek()Look at top element without removingO(1)
isEmpty()Check if emptyO(1)
size()Return number of elementsO(1)

In practice — Python

You don't implement stacks from scratch in interview problems. A Python list or deque works. The key difference:

Stack operationlistdequeComplexity
pushs.append(x)s.append(x)O(1) amort. / guaranteed
pops.pop()s.pop()O(1)
peeks[-1]s[-1]O(1)
isEmptylen(s) == 0len(s) == 0O(1)
list vs deque: list.append is O(1) amortized (can resize). deque.append is O(1) guaranteed — backed by a doubly-linked list of fixed-size blocks, no resizing. For stack-only use, list is fine. Use deque when you also need queue operations (append/pop from both ends).

If asked to implement from scratch

class Stack:
    def __init__(self):
        self._data = []

    def push(self, x):
        self._data.append(x)            # O(1) amortized

    def pop(self):
        if self.is_empty():
            raise IndexError("pop from empty stack")
        return self._data.pop()          # O(1)

    def peek(self):
        if self.is_empty():
            raise IndexError("peek from empty stack")
        return self._data[-1]            # O(1)

    def is_empty(self):
        return len(self._data) == 0

    def size(self):
        return len(self._data)

When to reach for a stack — common patterns

These problem signatures should immediately suggest a stack:

Bracket / parenthesis matching
Push opens, pop and verify on close. Classic ()[]{} validity check.
Next greater / smaller element
Monotonic stack: maintain a stack of candidates, pop when current element beats them.
Tracking running min/max
Use a second "min stack" alongside the main stack. Each entry stores (val, current_min).
Undo / backtrack state
Browser history, text editor undo — push state, pop to go back.
Expression evaluation
Infix → postfix, evaluate RPN. Operators and operands pushed/popped in order.
DFS iterative
Replace the call stack with an explicit stack when doing iterative depth-first traversal of graphs/trees.
Mental Model — Stack
  • Only the top matters. You never reach into the middle.
  • If a problem needs you to remember "what came before" and reverse it, think stack.
  • LIFO = the last thing you pushed is the first thing you deal with — very natural for nested structures (brackets, recursive calls, DFS).

Linear Data StructureLinked List

A linked list is a sequence of nodes. Each node is an object that holds two things:

Critical distinction: node.next is a reference to another node object — not to a value, not to a pointer. When you write curr = curr.next, curr becomes the next node, and you now have access to that node's .val and .next. Confusing nodes with values causes bugs in almost every linked list problem.

Unlike arrays, nodes can live anywhere in memory — no contiguous block required. The pointer is what chains them together.

Traversal

Use a curr pointer and advance with curr = curr.next. The golden rule:

Always check before you dereference. Before accessing curr.next, check curr is not None. Before accessing curr.next.next, check both curr and curr.next exist. This is the source of most linked list bugs. 
# Standard traversal
curr = head
while curr:               # checks curr != None
    # do something with curr.val
    curr = curr.next

# When you need .next.next — check both
while curr and curr.next:
    curr = curr.next.next

Head and tail

head = the first node object. tail = the last node object. They are node references, not values. tail.next = None always — that's how you know you've reached the end.

Inserting a node

If you know the position, insertion itself is O(1) — just redirect pointers. The rule to never forget:

Hold before you break. Before reassigning any pointer that you'll still need, save it in a temp variable. Then reconnect.

Removing a node

General removal: node.next = node.next.next — skip over the target. But finding the node costs O(n). The exceptions:

PositionComplexityWhy
HeadO(1)Just move head = head.next
Tail (singly linked)O(n)Must traverse to find prev node
Middle (given node)O(1)Pointer redirect only — but finding it is O(n)
Mental Model — Linked List
  • node is an object. node.val is the value. node.next is another node — never mix these up in your head.
  • Hold before you break — if reassigning a pointer you still need, save it first.
  • Always guard: curr before curr.next, and curr.next before curr.next.next.
  • Think from head and tail — not index. Access by index requires full traversal (O(n)).
  • tail.next is always None — that's your termination signal.

Linear Data StructureDoubly Linked List

Each node now has two pointers: prev and next — pointing to the node before and after it. Traversal in both directions becomes trivial.

Same guard applies: check curr before curr.prev, just like you check before curr.next.

Operations — pointer update order matters

Append to tail:

tail.next    = new_node      # 1. connect forward
new_node.prev = tail         # 2. connect backward
tail         = new_node      # 3. advance tail
# new_node.next is None by default ✓

Remove tail:

prev_node        = tail.prev  # 1. grab the node before tail
prev_node.next   = None       # 2. sever forward link
tail             = prev_node  # 3. move tail back

Both are O(1) — which is exactly what a stack needs. So a doubly linked list is a valid stack implementation, but dynamic arrays are preferred because they also give O(1) random access.

ComparisonArray vs Linked List

OperationArrayLinked List (singly)DLL
Access i-th element O(1) ▲ array wins O(n) O(n)
Insert / remove at end O(1)* O(n)† O(1) ▲ DLL wins
Insert / remove at head O(n) (shift all) O(1) ▲ LL wins O(1) ▲ DLL wins
Insert / remove middle (given position) O(n) (shift) O(n)‡ ▲ LL wins O(n)‡ ▲ LL wins

* dynamic array, amortized  † singly linked: O(n) to reach tail  ‡ pointer redirect is O(1) but traversal to reach middle is O(n). Array requires O(n) even with direct reference — due to shifting.

The real difference: for arrays the O(n) in middle insert/remove comes from shifting elements. For linked lists it comes from traversal to find the node. If you already have a reference to the node, linked list insert/remove is O(1). Arrays are O(n) regardless — you still have to shift.
Array vs Linked List — mental model
  • Arrays → think index. Random access is the superpower. Use when you need to jump around by position.
  • Linked lists → think head and tail. All operations radiate outward from the ends. Middle access is expensive.
  • In practice, arrays dominate interview problems because of O(1) index access. Linked lists appear when the problem is naturally pointer-based (merging, reversing, cycle detection).

Classic ProblemReversing a Linked List

Know this cold. First, the visual — see exactly which links break and reverse:

Iterative — 3-pointer dance

Keep three pointers: prev, curr, next_node. At each step, flip the arrow and advance all three.

def reverse_list(head):
    prev = None
    curr = head

    while curr:
        next_node  = curr.next   # 1. save next before breaking link
        curr.next  = prev        # 2. reverse the arrow
        prev       = curr        # 3. advance prev
        curr       = next_node   # 4. advance curr

    return prev              # prev is the new head

Recursive — the crux

The recursive insight: for any node head, assume the rest of the list is already reversed. All you need to do is make head.next point back at head:

Core of the recursion: head.next.next = head — make the node after me point back to me. Then set head.next = None to sever the original forward link. Repeat this going back up the call stack and the whole list reverses. 
def reverse_list(head):
    # base case: empty or single node
    if not head or not head.next:
        return head

    new_head = reverse_list(head.next)   # recurse to end; new_head = node 7

    head.next.next = head   # the node after head now points BACK to head
    head.next      = None  # sever head's original forward link

    return new_head         # bubble the new head all the way up

The call stack does the work — by the time recursion bottoms out at node 7 and starts returning, each frame executes head.next.next = head in reverse order, flipping all links.

Mental Model — Reversal
  • Iterative: you're physically walking the list and flipping each arrow as you go. Three pointers so you don't lose your place.
  • Recursive: "assume everything after me is already reversed — now make my successor point back at me." The base case is a single node (already reversed).
  • Both approaches return the new head (the old tail).
