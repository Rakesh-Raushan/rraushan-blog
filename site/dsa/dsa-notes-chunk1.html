Personal Notes — Arrays, Memory & Complexity

FoundationMemory & RAM

Data structures are fundamentally about how you arrange data in RAM — because RAM is limited and expensive, the way you structure data directly determines how efficiently you can use it.

Why "Random Access"? Given any address in RAM, you can jump to it directly in constant time — no scanning from the beginning. That's the "random" part, and it's why RAM is fast.

RAM is measured in bytes. A byte = 8 bits. A bit is a single position that holds either 0 or 1 — the language of computers.

In summary: bits → bytes → RAM.

Core ConceptArrays in RAM

The most important property of arrays: elements are stored as a contiguous block in RAM. This one fact drives everything about array performance.

Because of contiguous storage, you can jump to any element by arithmetic alone — no scanning needed:

address of arr[i] = start_address + bytes_per_element × i

For int arrays: +4 per step. For char arrays: +1 per step (ASCII = 1 byte).

Data StructureStatic Arrays

ReadO(1) — address arithmetic gives you any element directly.
Write (end)O(1) — same logic, you know where the end is.
DeleteO(n) worst case — deleting from the middle breaks contiguity, so you must shift up to n-1 elements to close the gap.
InsertO(n) worst case — inserting in the middle requires shifting elements to make room. Insert at end is O(1).
Python note: when you delete an element, Python removes the reference and marks it for garbage collection. The element doesn't vanish instantly — the GC handles cleanup later.

Data StructureDynamic Arrays

Python lists, JS arrays, Java's ArrayList, C++'s vector — all dynamic arrays. They solve the fixed-size problem of static arrays.

The trick is a second pointer marking the end of used space. When start reaches end (array is full), it resizes:

  1. Allocate a bigger block of memory
  2. Copy all elements to the new block
  3. Release the old block
How much bigger? Double the current size. Too small → frequent resizes. Too big → wasted memory. Doubling is the sweet spot — and the reason is Amortized O(1), explained below.

Amortized O(1) — the key insight

The cost of each append isn't constant — some appends trigger a copy. But averaged over many appends, it works out to O(1). This is called amortized complexity.

Why it stays bounded: by the time you reach size n, the total copies made = n/2 + n/4 + n/8 + … ≤ n. So total ops ≤ 2n, meaning average = 2n/n = O(1) amortized. The expensive resize events are rare enough that they don't drag the average up.
Doubling → O(1) amortized

Each resize doubles size → copies halve in frequency. Geometric series sums to O(n) total, O(1) per append average.

+1 growth → O(n) amortized

If you only grew by 1 each time, you'd copy on every single append. n appends → n copies on step n → O(n) copies → O(n²) total → O(n) per append. Bad.

FundamentalsTime Complexity

Complexity is about rate of growth, not exact time. We care about behaviour as n gets large — because that's where differences become real.

The question to ask: if I plot two functions, will one eventually overtake the other as n → ∞? If yes, the one that grows faster has worse complexity, regardless of constants or intercepts.

This is why we ignore constants: O(3n) and O(n) are parallel lines — one is just shifted. They both grow linearly, and neither will ever catch the quadratic O(n²). The class of growth is what matters.

Same logic for ignoring additive constants like O(n + 5) — two lines with different intercepts still diverge at the same rate. At large n, the +5 is noise.

Rule of thumb: we suppress constants and low-order terms because for any two functions with different growth classes, there always exists some n beyond which the faster-growing one dominates — it's certain, just a matter of when.
