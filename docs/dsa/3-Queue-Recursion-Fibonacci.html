Chunk 3 — Queue · Recursion · Fibonacci

Linear Data StructureQueue

Same idea as a stack but the order is reversed: FIFO — First In, First Out. The first element you add is the first one you get back. Think of a literal queue of people.

Two core operations: enqueue (add to the back) and dequeue (remove from the front). Because these happen at different ends, a simple array makes this awkward — dequeuing from the front requires shifting all elements, which is O(n). A doubly linked list handles both ends in O(1).

Why linked list, not array

The problem with an array: dequeue removes from the front, which requires shifting every element left — O(n). You could use an index to fake the front position, but that wastes memory and complicates the logic.

A doubly linked list makes both ends O(1): the head pointer is the front (dequeue = move head forward), the tail pointer is the back (enqueue = append new tail). No shifting, no wasted slots.

Python — always use deque

from collections import deque

q = deque()
q.append(x)        # enqueue — add to right/back    O(1) guaranteed
q.popleft()        # dequeue — remove from left/front O(1) guaranteed
q[0]              # peek front without removing
q[-1]             # peek back
len(q) == 0       # isEmpty check
Why not list? list.pop(0) — removing from the front of a list — is O(n) because every element shifts. deque.popleft() is O(1) guaranteed. This is the entire reason to prefer deque for queues.

Mental Model — Queue
  • Stack = LIFO (undo history, DFS). Queue = FIFO (processing order, BFS). When the order things were added matters, use a queue.
  • Always deque in Python — list.pop(0) is a silent O(n) trap.
  • Queues appear constantly in BFS, level-order traversal, and sliding window problems (monotonic deque).

Core ConceptRecursion

Recursion is when a function calls itself as part of solving its own problem. The mental overhead comes from trying to trace every call in your head — which you should never do. The mental models below are the alternative.

Building intuition — Factorial

Factorial is defined as: F(n) = n × (n−1) × (n−2) × ... × 1

You can loop through this trivially. But look at it differently: F(n) = n × F(n−1). The problem contains a smaller version of itself.

Insight 1 — Subproblem: can I express the solution to F(n) in terms of a solution to a smaller input? If yes, recursion is a candidate. F(n) = n × F(n−1) — yes.
Insight 2 — Base case: what is the smallest subproblem with a trivially known answer? This is where the recursion must stop. For factorial: F(1) = 1. Without a base case, the function calls itself forever.
General recursive framework
def rec_func(x):
    # 1. base case — stop recursing, return trivial answer
    if <smallest subproblem>:
        return <trivial answer>

    # 2. recursive case — call self on smaller subproblem
    return <combine> + rec_func(<smaller x>)
# Factorial — clean recursive implementation
def factorial(n):
    if n <= 1:              # base case
        return 1
    return n * factorial(n - 1)   # subproblem

Recursion has hidden space cost — the call stack

Factorial looks like it uses no extra memory — there's no array, no extra variable. But every function call is a stack frame held in memory until it returns. At the deepest point, all n calls are live simultaneously.

Recursive factorial vs iterative: time complexity is the same — O(n). But iterative uses O(1) space. Recursive uses O(n) space due to the call stack. No win here for recursion — its value is in problems where the recursive structure is genuinely cleaner (trees, divide-and-conquer, linked list reversal).
Mental Model — Recursion
  • Never trace the full call stack in your head. Trust the recursion: assume rec_func(smaller_input) works correctly, and just figure out how to use that result to solve the current input.
  • Find the subproblem (how to shrink the input), then find the base case (trivial stopping point). That's the whole design.
  • Recursion always has O(n) space on the call stack at minimum — it's not free even if you see no extra variables.
  • Good fit: tree traversal, linked list operations, divide-and-conquer. Bad fit: simple loops where iteration is equally clear.

Recursion — Two BranchFibonacci

Fibonacci: F(n) = F(n−1) + F(n−2), with base cases F(0) = 0, F(1) = 1.

Same two insights: there's a subproblem structure, and there are base cases. The difference: two subproblems per call instead of one. This changes everything about complexity.

def fib(n):
    if n <= 1:                         # base cases: F(0)=0, F(1)=1
        return n
    return fib(n - 1) + fib(n - 2)   # two branches

Why this is O(2ⁿ)

Look at the tree. Each level doubles the number of nodes. The tree has height n (because each call reduces the input by at least 1 before hitting the base case). So:

Level 0: 1 node (2⁰)  ·  Level 1: 2 nodes (2¹)  ·  Level 2: 4 nodes (2²)  ·  ...  ·  Level n: 2ⁿ nodes

Total nodes ≈ 2⁰ + 2¹ + 2² + ... + 2ⁿ = 2ⁿ⁺¹ − 1 ≈ O(2ⁿ)

Each node is one function call doing O(1) work. So total work = O(2ⁿ).

This is terrible for large n — 2³⁰ is over a billion. The fix is memoization (cache results so each subproblem is solved only once), which is the gateway to dynamic programming. But that's for later.

Mental Model — Two-branch recursion
  • One branch per call → O(n) calls total (linear tree). Two branches per call → O(2ⁿ) calls total (the tree doubles every level).
  • Tree height = how many times you can reduce the input before hitting the base case. Tree width = how many branches each node has. Total nodes = width^height.
  • Whenever you see two recursive calls — fib, tree traversal, subset generation — immediately think: this is exponential unless we memoize or prune.
