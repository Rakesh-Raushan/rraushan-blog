Sorting Algorithms

Ideas, implementations, complexity & when to use what

O(n²) · In-place · StableInsertion Sort

Core idea: a single-element array is always sorted. A sorted array of size n can absorb one more element and stay sorted — you just insert it at the right position. Build on exactly this: treat the leftmost element as a sorted array of size 1, then expand right one element at a time, each time inserting the new element into its correct position in the already-sorted left portion.

Pseudo code

# outer loop: i walks right, arr[0..i-1] is always sorted
for i in range(1, len(arr)):
    j = i - 1                    # start just left of i
    while j >= 0 and arr[j] > arr[j+1]:
        arr[j], arr[j+1] = arr[j+1], arr[j]   # swap left
        j -= 1

Best case O(n): if the array is already sorted, the inner while never executes. This is why insertion sort is used as the finishing step inside Timsort — it's exceptionally fast on nearly-sorted runs.
When to think about it
  • Almost never implement this in an interview — but interviewers occasionally ask you to explain it.
  • Know it because it's the clearest illustration of "grow a sorted sub-array by one at a time."
  • Used internally by Timsort (Python's built-in) for small sub-arrays (<64 elements) — that's its practical relevance.

O(n log n) · Not in-place · Stable · RecursiveMerge Sort

Two ideas combined: (1) a single-element array is sorted. (2) two sorted arrays can be merged into one sorted array in O(n) using three pointers. Recursively split the array until you have single elements, then merge back up.

The merge step uses three pointers: one for each input array and one for the output. Each comparison picks the smaller of the two current minimums — O(n) total per merge level.

Implementation — pass indices, not copies

Always work with indices. Slicing arrays to pass into recursive calls creates O(n) copies at every level — that's O(n log n) extra space for nothing. Pass start, mid, end indices and operate on the same array in-place. 
def merge(arr, start, mid, end):
    left  = arr[start:mid+1]    # temp copies only for the merge step
    right = arr[mid+1:end+1]
    i = j = 0
    k = start
    while i < len(left) and j < len(right):
        if left[i] <= right[j]:
            arr[k] = left[i]; i += 1
        else:
            arr[k] = right[j]; j += 1
        k += 1
    while i < len(left):  arr[k] = left[i];  i += 1; k += 1
    while j < len(right): arr[k] = right[j]; j += 1; k += 1

def merge_sort(arr, start, end):
    if start >= end:             # base case: single element
        return
    mid = (start + end) // 2
    merge_sort(arr, start, mid)
    merge_sort(arr, mid+1, end)
    merge(arr, start, mid, end)

# call: merge_sort(arr, 0, len(arr)-1)
all casesO(n log n) spaceO(n) — temp arrays stableyes adaptiveno external sortyes — disk-friendly
External sort winner: merge sort is the standard algorithm for sorting data that doesn't fit in RAM (files, databases). You sort chunks that fit in memory, write them to disk, then merge. No other algorithm here supports this cleanly.
Merge k sorted arrays/lists: the same merge idea extends naturally. For k=2 arrays, compare each pair — fine. For k arrays, comparing pairwise is O(k) per element. Instead, use a min-heap of size k seeded with the first element of each array. Pop the minimum (O(log k)), push the next element from that array (O(log k)). Total: O(n log k). The heap insight generalises: whenever you need repeated min/max from multiple sources, think heap.
When to think about it
  • Default choice when you need guaranteed O(n log n) — no worst-case degradation unlike quicksort.
  • Only O(n log n) sort that works well on linked lists (no random access needed for merging).
  • Needed for external sorting (data larger than RAM).
  • Pattern to recognise: "merge k sorted ___" → merge sort + heap.

O(n log n) avg · In-place · Unstable · RecursiveQuick Sort

Core insight: an element is in its final sorted position when everything to its left is smaller and everything to its right is larger. Quick sort finds that position for one element (the pivot), then recurses on the left and right sub-arrays.

The process of rearranging elements around the pivot is called partition. Pick the rightmost element as pivot. Use two pointers: left (tracks boundary of "smaller than pivot" region) and i (scans left to right). Whenever arr[i] < pivot, swap it to the left region and advance left. At the end, swap pivot into left position — that's its final sorted index.

def partition(arr, low, high):
    pivot = arr[high]
    left  = low - 1               # right boundary of "smaller" region
    for i in range(low, high):
        if arr[i] < pivot:
            left += 1
            arr[left], arr[i] = arr[i], arr[left]
    arr[left+1], arr[high] = arr[high], arr[left+1]  # pivot to sorted pos
    return left + 1              # pivot's final index

def quick_sort(arr, low, high):
    if low >= high:
        return
    pivot_idx = partition(arr, low, high)
    quick_sort(arr, low, pivot_idx - 1)
    quick_sort(arr, pivot_idx + 1, high)
avgO(n log n) worstO(n²) sorted input spaceO(log n) stack stableno adaptiveno
Why faster than merge sort in practice: both are O(n log n) average, but quicksort operates entirely in-place with sequential memory access. Merge sort writes to auxiliary arrays, causing cache misses. Modern CPUs prefetch sequential reads — quicksort's linear scan through the partition is extremely cache-friendly. Heapsort is also O(n log n) but jumps around memory even more — that's why quicksort beats heapsort in real benchmarks despite similar asymptotic complexity.
Worst case: already-sorted (or reverse-sorted) input with rightmost pivot gives O(n²) — pivot is always the min/max, one partition has size 0. Fix: random pivot, or median-of-three. Python's sort avoids this entirely.
When to think about it
  • Best general-purpose in-place sort for random data. Python/C++ standard libraries use quicksort variants internally.
  • Quickselect (find k-th smallest) uses the same partition idea — O(n) average. Important interview pattern.
  • Avoid on nearly-sorted data without randomised pivot.

O(n + k) · Not comparative · Linear timeBucket Sort (also: counting sort)

Core insight: if you know the value range, you don't need to compare elements at all. Scan the array once to build a frequency map (one slot per possible value), then reconstruct the sorted array by walking the slots in order.

def bucket_sort(arr):
    if not arr: return arr
    min_v, max_v = min(arr), max(arr)
    buckets = [0] * (max_v - min_v + 1)

    for v in arr:                 # build freq map — O(n)
        buckets[v - min_v] += 1

    result = []
    for i, count in enumerate(buckets):   # reconstruct — O(k)
        result.extend([i + min_v] * count)
    return result
timeO(n + k) spaceO(k) buckets stableyes comparativeno
Disguised bucket sort problems: the sort is often hidden. E.g. — 1000-item list but values are only R/G/B (Dutch flag problem), or sort by age (0–120), or top-k frequent elements. Whenever the domain of values is bounded and small relative to list size — think buckets.
When to think about it
  • Values have a known, small range relative to the list size — the range k should feel cheap.
  • The problem says "sort" and the elements can only take limited distinct values.
  • Also known as counting sort — same idea, different name.

O(n · d) · Not comparative · StableRadix Sort

The magic trick: instead of comparing elements, sort by individual digits — ones place first, then tens, then hundreds, all the way up. After d passes (d = number of digits in the max element), the array is sorted.

Why does it work? Because each single-digit pass is a stable sort (preserving relative order of ties), and starting from the least significant digit means more significant digits correctly override when needed.

def counting_sort_by_digit(arr, exp):
    n = len(arr)
    output  = [0] * n
    buckets = [0] * 10
    for v in arr:
        buckets[(v // exp) % 10] += 1
    for i in range(1, 10):           # prefix sums → positions
        buckets[i] += buckets[i-1]
    for i in range(n-1, -1, -1):    # fill right-to-left for stability
        idx = (arr[i] // exp) % 10
        output[buckets[idx]-1] = arr[i]
        buckets[idx] -= 1
    arr[:] = output

def radix_sort(arr):
    max_v = max(arr)
    exp = 1
    while max_v // exp > 0:
        counting_sort_by_digit(arr, exp)
        exp *= 10
timeO(n · d) spaceO(n + b) b=base stableyes comparativeno
Why radix is incredibly fast on modern CPUs: when the base is a power of 2 (e.g. base 256), extracting a digit becomes a bitmask + shift — (v >> 8) & 0xFF. No division, no modulo. For 32-bit integers with base 256, d=4 passes over the array. Cache-friendly sequential scans, no branching from comparisons. Radix sort on integers often beats O(n log n) comparison sorts in wall-clock time despite the same or worse asymptotic complexity.
Sorting strings: radix sort works naturally on fixed-length strings — sort by last character, then second-to-last, etc. For variable-length strings, you can pad or use MSD (most-significant-digit) radix sort.
When to think about it
  • Large value range but small list — bucket/counting sort would need huge arrays; radix handles it with d passes.
  • Sorting integers or fixed-length strings at maximum speed.
  • Opposite suitability from bucket: bucket = spread specialist, radix = depth specialist.

When to use what
General purpose / default
Just use arr.sort() — Timsort, O(n log n) guaranteed, stable, adaptive. Don't overthink it.
Need guaranteed O(n log n) + stability
Merge sort. Timsort also qualifies. Pick merge sort if you need external or linked-list sort.
In-place, fast on random data
Quick sort with random pivot. Best cache behaviour. Avoid on nearly-sorted without randomisation.
Data is mostly sorted already
Insertion sort (O(n) best case) or Timsort (recognises natural runs automatically).
Integers / chars, small value range
Bucket / counting sort. O(n) time. Especially good when values are categorical (colours, ages, grades).
Integers, large range but small list
Radix sort. d passes of O(n), range doesn't matter. Bitmask version is fastest for 32/64-bit ints.
Sorting a stream (elements arrive one at a time)
Insertion sort (online). Or a heap to maintain top-k. Merge/quick/radix all need the full array first.
Data too large for RAM
Merge sort. Sort RAM-sized chunks, write to disk, merge. Only sort suitable for external data.
Sorting fixed-length strings
Radix sort character by character (LSD). Extremely fast — no comparisons, just bucket passes.
Interview: "implement a sort"
Merge sort or quick sort. Know both cold. Merge if they want stable/guaranteed; quick if they want in-place.
